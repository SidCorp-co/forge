## Top-line decision

**Recommendation: cut `v0.1.0-rc.6` — do NOT promote rc.5 to stable `v0.1.0`.**

rc.5 is a major step forward from rc.3 — every rc.3 hard blocker is verifiably fixed:

1. ✅ Auth surface restored: real public landing at `/`, working `/login` form (with proper "invalid credentials" error and rate-limit defense), working `/register`, real user identity in the sidebar pill (no more "Agent User" spoof), working logout. (ISS-239)
2. ✅ `forge_auth` cookie carries `Secure; HttpOnly; SameSite=Lax` again. (ISS-240)
3. ✅ `/ws` reachable through Cloudflare edge — handshake returns 101 Switching Protocols with auth cookie; browser `WebSocket` opens cleanly to `wss://`. (ISS-238)
4. ✅ Issue list rows are `<a>` links — clicking row navigates to issue detail under `/projects/:slug/issues/ISS-N`. (ISS-241)
5. ✅ Kanban renders existing seed issues in their correct columns. (ISS-242)
6. ✅ Custom 404 page; 401 / cleared-cookie now redirects to `/login` cleanly. (ISS-243)
7. ✅ XSS hardening confirmed — `<script>` and `<img onerror=>` payloads in title/description render as escaped text, no script execution.

But rc.5 has **two material regressions / gaps** that block calling it stable:

- ❌ **Kanban is missing the `Closed` column.** The pipeline status enum has 16 values (`open, confirmed, waiting, approved, in_progress, developed, deploying, testing, tested, pass, staging, released, closed, reopen, on_hold, needs_info`). The board renders 15: `Open, Confirmed, Waiting, Approved, In Progress, Developed, Deploying, Testing, Tested, Pass, Staging, Released, Reopen, On Hold, Needs Info` — `Closed` is gone. Closed issues will silently disappear from the board, which is a release-blocker for a project tool whose primary surface is the kanban.
- ❌ **Issue detail has no editable surfaces beyond "Close issue".** The page renders title, status badge, priority badge, description, empty `Comments` and `Activity` sections — and a banner that explicitly says "rich view is not available yet". There is no status transition dropdown, no add-comment editor, no add-label control, no assignee picker, and no markdown rendering for the description (literal `**bold**` shows as text). The pipeline can be operated by API but the cloud UI cannot drive an issue end-to-end. v0.1.0 stable should not ship without at least the comment editor and status transition.

Plus three lower-severity items that are independently filed: register form has a contrast/styling issue (button text is white-on-white when idle), kanban cards do not show the `displayId` (ISS-N), and the issues list has no filter/search controls.

Details below.

## Decision rule applied

- All deep-walk sections (0, 1, 2, 5, 6, 7, 10) ✅ AND no spot-check regression → recommend tag `v0.1.0` stable.
- Any deep-walk section ❌ on a rc.3 carry-over OR any new regression on a previously-passing section → recommend cut `v0.1.0-rc.6`.

**Sections 5 and 6 are deep-walk ❌** for new gaps that are independent of the rc.3 fix scope: missing `Closed` column on the kanban, and an essentially read-only issue detail. Those are not regressions vs rc.3 (rc.3 was worse on both counts), but they are blockers against calling rc.5 the stable v0.1.0 cut.

→ **Cut v0.1.0-rc.6** with the high-severity bugs above addressed (re-add `Closed` column to the kanban; ship the status-transition + comment-editor minimum on issue detail). The remaining medium/low items can ride along or defer.
