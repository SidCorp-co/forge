## Top-line decision

**Recommendation: cut `v0.1.0-rc.3` — do not promote rc.2 to stable `v0.1.0`.**

Two hard blockers surfaced before the happy-path could be executed end-to-end on this deployment, and one major UX gap was observed. Neither blocker is in "Section 9 LiteLLM" territory (i.e. neither is acceptable to skip for the stable tag per the plan's decision rule):

1. **`/ws` is not reachable from the public edge.** `GET /wss://stg-jarvis-a2.thejunix.com/ws` with a proper WebSocket upgrade handshake returns **HTTP 404** with the core JSON 404 body. `/health` reports `ws: ok`, so the WS server is up inside the container — the Cloudflare/edge/reverse-proxy layer is not routing `/ws` to the core process. This means every WS-backed behaviour (Section 6 real-time, Section 7 job dispatch, agent streams) is dead from a browser/device over the internet.
2. **Staging has no way to complete email verification.** All authenticated endpoints are gated by `assertEmailVerified()` and return `{code:"EMAIL_NOT_VERIFIED"}`. SMTP is not configured on this deployment, there is no admin/dev "force-verify" endpoint, and the verification email is only written to the logger when SMTP is absent (`forge/core/src/auth/email.ts:29`). With no log/DB access from this tester surface, Sections 3–5 & 7 cannot be executed through the API at all against this staging without operator-side intervention. The plan anticipated this risk; the gap is that the staging env was not prepared with either SMTP, a seeded verified user, or a documented "manual-verify via DB" recipe that the tester can execute unassisted.
3. **Web UI has no project-list route.** `GET /projects` from the browser returns Next.js 404, and `/dashboard` only renders the `UnimplementedBanner` "Dashboard is not available yet" card. There is no navigation surface in the logged-in UI that lets a new user create a first project — the features at `/projects/[slug]/*` exist but are unreachable from the nav after login. This is a v0.1 UX regression independent of the verification blocker.

Details, evidence, and per-item pass/fail below. A new issue should be opened (or the work absorbed into ISS-235) for each of the three findings; see **Bugs / follow-ups** at the bottom.

## Bugs / follow-ups

Each of these should be filed as a Forge issue linked back to ISS-236, or absorbed into ISS-235 if it fits the Phase 2.9-T1 tech-debt scope.

1. **`/ws` unreachable through the edge on staging** — Cloudflare/nginx is returning 404 for WebSocket upgrade at `/ws`. Core listens on `/ws` and `/health` reports `ws:ok`. Fix at the reverse-proxy / Coolify ingress config, then re-run Section 6. *(Priority: high — release blocker.)*
2. **Staging has no usable email-verification path** — SMTP not configured and no dev-skip endpoint exists. Either (a) wire SMTP for the staging deploy, (b) add a guarded-by-env `POST /api/auth/force-verify` endpoint for `NODE_ENV !== 'production'`, or (c) seed a pre-verified test user and expose its creds to QA. Document whichever in the runbook so future E2E can proceed. *(Priority: high — release blocker for QA-by-tester.)*
3. **Web UI missing `/projects` list page + nav** — `/projects` returns Next 404 and `/dashboard` shows `UnimplementedBanner` only. Users have no way to discover or create a first project through the UI after login. Check whether the list route was deleted during the 2.9 refactor or simply never reconnected to the nav. *(Priority: high for UX.)*
4. **Auth rate-limit spec drift** — plan assumed `/api/auth/local` allows 10/min/IP but the live config returns `x-ratelimit-limit: 5`. Reconcile `forge/core/src/config/rate-limits.ts` with the documented contract or update the docs. *(Priority: low.)*
5. **MCP guidance** — `POST /mcp` returning "invalid device token" for a **user** JWT is correct behaviour but not obvious. Consider returning a more specific hint (`AUTH_REQUIRES_DEVICE_TOKEN`) and/or documenting in the MCP section of the module docs that MCP is device-scoped. *(Priority: low.)*
