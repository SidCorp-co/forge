## Top-line decision

**Recommendation: cut `v0.1.0-rc.4` вЂ” do NOT promote rc.3 to stable `v0.1.0`.**

rc.3 fixed exactly **one** of the three rc.2 hard blockers (the `/projects` web route is now reachable and the seeded project lists). The other two blockers are still in place AND rc.3 introduces a new class of regressions on top:

1. **`/ws` is still 404 through the Cloudflare edge.** Identical symptom and response shape as rc.2 вЂ” `/health` reports `ws:ok` inside the container but the public WSS upgrade returns `{"code":"NOT_FOUND","message":"Not Found: GET /ws"}`. Every Section 7 (real-time) check is dead.
2. **Web app has no auth wall and renders an unauthenticated user as a logged-in identity.** With cleared cookies/localStorage, hitting `/`, `/login`, `/register`, `/dashboard`, `/projects`, `/projects/<slug>/*`, `/pipeline` all return 200 with the full sidebar shell, a hardcoded **"Agent User В· System Online"** identity, and a "Connected" status pill. There is no `/login` page anymore вЂ” `/login` 200s into the same shell, and `/auth/login` 404s. The backend still rejects API writes with `401` (so data is not actually leaking), but the UX gives a misleading impression that the user is signed in. This is a v0.1 release blocker on multiple axes (security perception, register/login flows, "Forgot password", logout discoverability вЂ” all of those tasks have no surface to execute).
3. **`forge_auth` cookie no longer carries the `Secure` flag.** rc.2 set `forge_auth=...; Max-Age=604800; Path=/; HttpOnly; Secure; SameSite=Lax`. rc.3 sets `...; HttpOnly; SameSite=Lax` only вЂ” `Secure` is gone. Cookie is still served over HTTPS but a misconfigured proxy or accidental HTTP redirect will now leak it.
4. **Issue list rows are not interactive вЂ” there is no way to open an issue from the UI.** The list renders ISS-1 cleanly but the row is not an `<a>`, has no onClick, no role, and `cursor: auto`. Visiting `/projects/smoke-rc3/issues/ISS-1` directly returns "INVALID INPUT вЂ” please check the fields and try again" (the page expects a documentId, not the displayId, but nothing in the UI hands you a documentId).
5. **Front-end still issues Strapi-flavoured queries that the new core does not implement.** Every page run logs `404 /api/notifications/unread-count` and `404 /api/user-preferences?filters[userKey][$eq]=user:<uuid>` вЂ” left over from the pre-Strapi-removal cutover. The `?filters[...][$eq]=` query-string is the Strapi REST flavour, not Hono.

Rc.2 carried a third blocker вЂ” "no usable verification path" вЂ” which in rc.3 is **resolved by seeding** the `admin@thejunix.com` user with `emailVerified:true` (verified by API: `emailVerificationRequired: false` in the login response). That is acceptable as a staging workaround but the SMTP gap and dev-skip absence are still present for any new user; the email-verify gate UX in the browser still cannot be exercised because there is no register/login surface.

## Bugs / follow-ups

Each of these should be filed as a new Forge issue (titled `[rc.3 UX] ...`) linked back to ISS-237. Do **not** reopen ISS-235 (closed) or any rc.2 issue.

1. **`/ws` unreachable through the edge вЂ” UNCHANGED rc.2 BLOCKER.** Cloudflare/Coolify ingress still 404s the WSS upgrade at `/ws`. Core listens on `/ws` and `/health` reports `ws:ok`. Same fix as rc.2 follow-up #1. *(Priority: critical вЂ” release blocker.)*
2. **No login / register / logout UI surface in the web app.** `/login`, `/register`, `/auth/login` either render the wrong shell or 404. Anonymous users are dropped straight into a fake "Agent User" identity, and the bottom-left identity pill never reflects the real user. *(Priority: critical вЂ” release blocker; security-perception bug even though backend auth is intact.)*
3. **`forge_auth` cookie missing the `Secure` flag.** rc.2 had `Secure`; rc.3 dropped it. Set it back wherever the cookie is issued (`forge/core/src/auth/cookies.ts` or equivalent). *(Priority: high.)*
4. **Issues list rows are non-interactive; issue detail unreachable.** `/projects/<slug>/issues` row is not an `<a>`; visiting `/projects/<slug>/issues/<displayId>` returns "INVALID INPUT". The list must link to `/projects/<slug>/issues/<documentId>` (or the detail page must accept displayId). *(Priority: critical вЂ” release blocker; the entire issue detail surface is gated behind this.)*
5. **Kanban does not render existing approved issues.** Every column shows 0 even though ISS-1 is `approved`. Also has a "Draft" column that is not in the documented 14-status pipeline. *(Priority: high.)*
6. **`/api/notifications/unread-count` returns 404 on every page load.** Endpoint is missing on `forge/core`; the web client polls it on every navigation. Either implement the endpoint or remove the polling. *(Priority: medium.)*
7. **`/api/user-preferences?filters[userKey][$eq]=user:<uuid>` returns 404 on every page load.** Strapi-flavoured query string against a Hono backend that does not implement that route. Cutover leftover. *(Priority: medium.)*
8. **`/dashboard` renders without the sidebar shell.** Inconsistent with every other top-level nav target. *(Priority: low UX.)*
9. **Empty states are red errors instead of empty states.** "Failed to load projects." (red text) is shown when the API returns 401/403, with no recovery CTA. Replace with a real empty / sign-in state. *(Priority: medium.)*
10. **"New project" CTAs go nowhere.** Sidebar "+ New Project" links to `/dashboard` (banner), header "New Project" button has no handler. *(Priority: high вЂ” users cannot create a project from the UI.)*
11. **New-issue form silently no-ops on empty submit.** No inline error, no toast. *(Priority: medium.)*
12. **404 page is unstyled / ungranded.** Bare Next.js default. *(Priority: low.)*
13. **Auth rate-limit spec drift (carried from rc.2 #4).** Login rate limit observed at `x-ratelimit-limit: 5` per minute; original plan referenced 10/min. Reconcile docs vs code. *(Priority: low.)*
